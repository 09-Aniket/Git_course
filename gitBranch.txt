The Core Philosophy of Branching:
  In Git, a branch is not a heavy copy of your project's folders and files. Instead, a branch is a lightweight, movable pointer that tracks a specific commit [1].

  When you make a new commit, your active branch pointer automatically moves forward to that new commit [1]. This architecture makes creating and switching branches nearly instantaneous, regardless of project size.

The 4 Git States of Branching:
  To visualize how branches move, you must understand four key pointers:
    1.HEAD: A pointer that tells Git which branch you are currently working on.
    2.main / master: The default primary pointer representing your stable, production-ready code.
    3.Feature Branch: A temporary pointer created to isolate a specific task or bug fix.
    4.Remote Tracking Branch (e.g., origin/main): A read-only pointer representing the exact state of the branch on your remote server (like GitHub) during your last connection.

Phase 1: Basic Branching Operations
   Creating a Branch:
    This creates a new pointer referencing the exact commit your HEAD is currently on. It does not automatically switch you to that branch.
    Command: git branch <branch_name>
    Example: git branch feature-login

  Switching Branches: 
    This moves the HEAD pointer to the targeted branch and updates the files in your working directory to match that branch's last snapshot.
    Command: 
    git checkout <branch_name> 
    or the modern 
      git switch <branch_name>
    Example: git switch feature-login

  The Shortcut (Create & Switch)
    Creates the branch pointer and moves your HEAD to it simultaneously.
        Command: 
        git checkout -b <branch_name> (b-->branch)
        or 
        git switch -c <branch_name> (c-->create)
        or 
        git switch - (switch to previous branch)
        
        Example: git switch -c feature-login

  Listing Branches
    Local branches: git branch (The branch with the asterisk * is your active branch).
    All local and remote branches: git branch -a
    

Phase 2: Intermediate Merging & Conflict Resolution
   Once your work on a feature branch is stable, you bring those changes back into your primary branch.

    1. Fast-Forward Merge
        Occurs if the main branch has not received any new commits since you created your feature branch. Git simply slides the main pointer forward to match your feature branch pointer. No new merge commit is made.
        git checkout main
        git merge feature-login
        
    2.Three-Way Merge (No-Fast-Forward)
       Occurs if the main branch received other commits while you were working on your feature branch. Git looks at the common ancestor commit, the tip of main, and the tip of your feature branch, creating a brand new Merge Commit to bind them.
       git checkout main
       git merge feature-login
       # A text editor will open asking for a merge commit message. Save and exit to complete.

    3.Handling Merge Conflicts
       If you and a teammate edit the exact same line of the same file on different branches, Git cannot automatically choose which code to keep. The merge will pause, and Git will alter your files to show the conflict:
            <<<<<<< HEAD
            YOUR CHANGES ON MAIN BRANCH
            =======
            THEIR CHANGES ON FEATURE BRANCH
            >>>>>>> feature-login

        How to resolve it:
        1.Open the file in your code editor.
        2.Manually delete the conflict markers (<<<<<<<, =======, >>>>>>>) and edit the code to your liking.3.3.Stage the resolved file: git add <file_name>
        4.Finalize the merge: git commit -m "Resolve merge conflict"

    4.Deleting Branches
    Clean up your workspace after a successful merge.
        Delete merged branch safely: git branch -d feature-login
        Force delete unmerged branch: git branch -D feature-login (Warning: Loses unmerged work).

Phase 3: Advanced Branching Techniques
    Git Rebasing (Clean History)
     Instead of combining branches with a messy merge commit, rebasing takes all the commits made on your feature branch and re-applies ("plays") them line-by-line on top of the tip of the main branch. This creates a completely linear project history.
       git checkout feature-login
       git rebase main

    Golden Rule of Rebasing: Never rebase a branch that you have already pushed to a public remote repository, as it rewrites git history and disrupts your team's tracking pointers.

    Cherry-Picking:Allows you to select a single, specific commit from any branch and copy it onto your current active branch without merging the entire branch.
     Command: git cherry-pick <commit_hash>
     Example: git cherry-pick a1b2c3d

    Tracking Remote Branches:Link a local branch directly to a branch on GitHub so you can easily pull and push edits.
    Command: git branch --set-upstream-to=origin/<branch_name> <local_branch_name>

    Visualizing Branch History:To see your local pointer structure and merges in your terminal, use this comprehensive log configuration:
         git log --oneline --graph --all
